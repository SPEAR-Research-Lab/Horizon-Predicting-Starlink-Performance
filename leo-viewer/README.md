# LEO Viewer

Interactive map showing predicted Starlink latency and throughput globally. Uses H3 hexagonal grids with zoom-adaptive resolution.

Built as the deployment companion to the [Horizon paper](https://doi.org/10.1145/3805639) (SIGMETRICS 2026).

## Architecture

```
weekly-measurements-collection/    <- collects data (GitHub Actions, weekly)
train-predict-pipeline/            <- trains models + generates predictions (AWS EC2, weekly)
leo-viewer/frontend/               <- static Vue app consuming prediction JSONs
website/                           <- paper presentation page
```

The frontend is fully static. No backend needed. Prediction JSONs are pre-generated by the AWS pipeline and committed to `frontend/public/` on the `predictions-data` branch.

## Frontend

```bash
cd leo-viewer/frontend
npm install
npm run dev
```

Open http://localhost:5173

### Build for production

```bash
npm run build-only
# Outputs to dist/, serve with any static hosting
```

Production deployment is automated via GitHub Pages (see `.github/workflows/deploy-pages.yml`).

## Zoom-Adaptive Hexagons

The grid map adjusts hexagon resolution based on zoom level:
- Zoom < 4: H3 resolution 2 (~1,200 km hexagons)
- Zoom 4-6: H3 resolution 3 (~460 km hexagons)
- Zoom > 6: H3 resolution 4 (~175 km hexagons)

Higher-resolution data is lazy-loaded on demand.

## Views

- **Grid Map:** H3 hexagonal grid with color-coded network quality
- **Dot Map:** City-level point predictions

## Data Flow

1. `weekly-measurements-collection` collects Starlink measurements from M-Lab + Cloudflare (GitHub Actions, weekly)
2. `train-predict-pipeline` trains fresh models on that data, predicts for all hex centers (AWS EC2 spot)
3. Prediction JSONs get pushed to the `predictions-data` branch
4. GitHub Pages deployment pulls predictions and builds the frontend

## Tech Stack

- Vue 3 + TypeScript
- MapLibre GL (map rendering)
- h3-js (hexagon geometry)
- Turf.js (geospatial utilities)
- Vite (build tooling)
